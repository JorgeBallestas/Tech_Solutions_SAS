PRD - Tech Solutions SAS (Versión Core)

Versión: 1.1 | Fecha: 12 de Febrero de 2026

1. Introducción

El presente documento describe los requisitos para el desarrollo de una aplicación interna para Tech Solutions SAS. El objetivo principal del "Sistema de Gestión de RRHH" es centralizar y automatizar los procesos clave del departamento de recursos humanos.

Foco del Desarrollo: El desarrollo se realizará siguiendo estrictamente la metodología TDD y se centrará exclusivamente en la lógica de negocio del backend (modelos, controladores, etc.), sin desarrollar una interfaz gráfica de usuario.

2. Visión y Objetivos del Producto

Visión

Ser la plataforma backend única y confiable para toda la gestión del ciclo de vida de los contratos de los colaboradores de Tech Solutions SAS.

Objetivos
  • Automatizar la creación y gestión de contratos, prórrogas y terminaciones a través de una lógica de negocio bien definida y probada.
  • Garantizar la integridad y consistencia de los datos mediante pruebas automatizadas.
  • Establecer un control de acceso robusto basado en roles a nivel de API.

3. Metodología de Desarrollo: TDD

El proyecto debe basarse en Test-Driven Development (TDD). El ciclo de desarrollo para cada funcionalidad será:

1. Escribir Test que Falla
(ROJO)
2. Escribir Código para Pasar
(VERDE)
3. Refactorizar
(REFACTOR)

4. Alcance y Funcionalidades (Features)

La aplicación contará con los siguientes módulos de backend. A continuación se detallan las historias de usuario para cada uno:

Actor(es): RRHH.

Descripción: Permite crear, ver, actualizar y desactivar los perfiles de los colaboradores en el sistema.

Flujo Principal:

  1. El usuario accede al módulo "Colaboradores".
  2. El sistema muestra una lista de colaboradores existentes.
  3. El usuario puede (a) hacer clic en "Nuevo Colaborador" o (b) seleccionar uno existente para editarlo.
  4. (a) El sistema presenta un formulario para ingresar datos personales (nombres, apellidos, tipo y número de documento, fecha de nacimiento, etc.).
  5. (b) El sistema presenta el mismo formulario con los datos cargados.
  6. El usuario guarda los cambios.
  7. El sistema valida los datos y los persiste en la base de datos.

Actor(es): RRHH.

Descripción: Permite crear y administrar los contratos vinculados a un colaborador.

Flujo Principal:

  1. El usuario navega al perfil de un colaborador.
  2. Selecciona la pestaña "Contratos" y hace clic en "Nuevo Contrato".
  3. El sistema muestra un formulario para datos del contrato (tipo de contrato, fecha de inicio, fecha de fin, cargo, salario, etc.).
  4. El usuario completa y guarda el formulario.
  5. El sistema crea el registro del contrato y lo asocia al colaborador.

Actor(es): RRHH.

Descripción: Permite añadir una extensión de tiempo o una adición de valor a un contrato existente.

Flujo Principal:

  1. El usuario localiza el contrato activo al que desea añadir una prórroga.
  2. Hace clic en la opción "Añadir Prórroga/Adición".
  3. El sistema presenta un formulario para especificar el tipo (Tiempo/Valor), la nueva fecha de finalización (si aplica) y el valor de la adición (si aplica).
  4. El usuario guarda los cambios.
  5. El sistema registra la prórroga y actualiza el estado o valores del contrato principal si es necesario.

Actor(es): RRHH.

Descripción: Registra la finalización de un contrato antes de la fecha pactada.

Flujo Principal:

  1. El usuario localiza el contrato activo que desea terminar.
  2. Selecciona la opción "Terminación Anticipada".
  3. El sistema solicita la fecha de terminación y un campo para justificar el motivo.
  4. El usuario confirma la acción.
  5. El sistema actualiza el estado del contrato a "Terminado" y guarda la información de la terminación.

5. Roles y Permisos de Usuario

Rol Módulos Permitidos
Gestor RRHH Gestión de Colaboradores, Gestión de Contratos, Gestión de Prórrogas, Terminación de Contratos.

6. Casos de Prueba (para TDD)

  • Prueba: Verificar que se puede crear un nuevo colaborador con datos válidos.
  • Prueba: Verificar que el sistema rechaza la creación de un colaborador con un número de documento duplicado.
  • Prueba: Verificar que se puede actualizar la información de un colaborador existente.
  • Prueba: Verificar que se puede obtener el listado de todos los colaboradores.
  • Prueba: Verificar que se puede eliminar (o desactivar mediante soft-delete) un colaborador.

  • Prueba: Verificar que se puede crear un contrato y asociarlo a un colaborador existente.
  • Prueba: Verificar que no se puede crear un contrato para un colaborador inexistente.
  • Prueba: Verificar que los campos de fecha y salario son validados correctamente.
  • Prueba: Verificar que se puede actualizar un contrato existente.

  • Prueba: Verificar que se puede añadir una prórroga (en tiempo o valor) a un contrato de tipo "Fijo" o "Prestación de Servicios".
  • Prueba: Verificar que la fecha de finalización del contrato se actualiza correctamente al añadir una prórroga de tiempo.
  • Prueba: Verificar que el sistema rechaza una prórroga para un contrato con estado "Terminado" o "Finalizado".

  • Prueba: Verificar que se puede cambiar el estado de un contrato a "Terminado".
  • Prueba: Verificar que se registra correctamente la fecha y el motivo de la terminación.
  • Prueba: Verificar que no se puede terminar un contrato que ya ha finalizado.

7. Requisitos No Funcionales

Seguridad
  • Control de acceso basado en roles (RBAC) a nivel de API.
  • Contraseñas almacenadas con hash.
  • Prevención de ataques comunes (XSS, CSRF, SQL Injection).
Rendimiento
  • Consultas a la base de datos optimizadas.
  • Uso eficiente de Eloquent ORM.

9. Modelo de la Base de Datos

A continuación, se presenta un esquema simplificado de las tablas principales y sus relaciones.

users
(Gestionada por Laravel)
  • id (PK)
  • name
  • email (unique)
  • password
  • ... (timestamps, etc.)
collaborators
  • id (PK)
  • first_name
  • last_name
  • document_type (enum: CC, CE, PPT)
  • document_number (unique)
  • birth_date
  • email
  • phone_number
  • address
  • ... (timestamps)
contracts
  • id (PK)
  • collaborator_id (FK a collaborators.id)
  • contract_type (enum: Fijo, Indefinido, Prestación de Servicios)
  • start_date
  • end_date (nullable para indefinidos)
  • position (cargo)
  • salary (decimal)
  • status (enum: Activo, Terminado, Finalizado)
  • ... (timestamps)
contract_extensions
(Adiciones o Prórrogas)
  • id (PK)
  • contract_id (FK a contracts.id)
  • extension_type (enum: Tiempo, Valor)
  • new_end_date (nullable)
  • additional_value (decimal, nullable)
  • description
  • created_at
contract_terminations
  • id (PK)
  • contract_id (FK a contracts.id, one-to-one)
  • termination_date
  • reason (text)
  • created_at
Relaciones
  • Un Collaborator puede tener muchos Contracts.
  • Un Contract pertenece a un Collaborator.
  • Un Contract puede tener muchas Contract_Extensions.
  • Un Contract puede tener una Contract_Termination.
  • Un User tiene un rol (spatie/laravel-permission gestiona las tablas roles, permissions, model_has_roles).

10. Documentación y Entregables

Código Fuente y Git Flow

El código fuente deberá ser entregado en un repositorio de GitHub, siguiendo estrictamente el flujo de trabajo Git Flow:

  • main: Código de producción estable.
  • develop: Rama principal de desarrollo.
  • feature/*: Ramas para nuevas funcionalidades.
  • release/*: Ramas para preparar lanzamientos.
  • hotfix/*: Ramas para correcciones urgentes.
Conventional Commits

Los mensajes de los commits deben seguir esta especificación. Ejemplo:

feat(Collaborators): Implementa CRUD para Colaboradores con TDD
Entregables Finales
  • Repositorio de GitHub completo.
  • Documentación de casos de prueba.
  • Seeders para poblar la base de datos con datos de ejemplo.
